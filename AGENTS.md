# AGENTS.md

## Область действия и общение

Эти инструкции действуют для всего репозитория.

- Общайся с пользователем на русском языке.
- Начинай с результата. Пиши короткие понятные обновления и объясняй технические детали простым языком.
- Не выдумывай факты о репозитории, продакшене, Telegram API и внешних сервисах. Сначала изучи код, конфигурацию, тесты и при необходимости историю Git.
- Задавай вопрос только тогда, когда недостающий выбор существенно меняет результат или требуется новое разрешение пользователя.

## Проект

В репозитории находится «Незабудка» — Telegram-бот для напоминаний в личных и групповых чатах.

Основной стек:

- Python 3.14;
- aiogram;
- FastAPI;
- APScheduler;
- SQLite;
- Telegram Mini App в `tma/`;
- Docker Compose в продакшене;
- Nginx как внешний компонент production-инфраструктуры, конфигурация которого находится на VPS и не хранится в репозитории;
- Open-Meteo для погодных напоминаний.

Публичный HTTPS reverse proxy в продакшене — Nginx. Его фактическую конфигурацию
можно проверять только на VPS после прямого разрешения на доступ к продакшену.

Важные продуктовые ограничения:

- Mini App — основной пользовательский интерфейс. Не добавляй сценарии с длинными командами без прямого запроса пользователя.
- Напоминания и настройки принадлежат Telegram-чату, идентифицируемому по `chat_id`.
- Mini App не должна доверять `chat_id` от клиента. Сохраняй проверку Telegram `initData`, срока `auth_date`, подписанного launch token и соответствия запрошенного чата подписанному контексту.
- Работа с часовыми поясами должна оставаться явной и обратно совместимой. Историческое значение по умолчанию — `Asia/Yekaterinburg`; перед изменениями проверь текущую реализацию.
- Таблицы `reminders` и `chat_settings` содержат пользовательские данные. Таблицы `weather_location_cache` и `weather_report_cache` содержат производный кэш, который допустимо явно инвалидировать, если это не меняет напоминания и настройки и закреплено тестом.
- Сервис уже работает в продакшене. Сохраняй пользовательские данные и существующее поведение, если пользователь явно не согласовал ломающие изменения.
- Различай тестирование ПО и обеспечение качества. Не используй «QA» как синоним тестирования в пользовательских текстах и документации.

## Перед началом изменений

1. Выполни `git status --short` и сохрани все существовавшие до начала задачи изменения пользователя.
2. Изучи относящиеся к задаче реализацию, тесты и конфигурацию. При необходимости используй историю Git, чтобы понять причины текущего поведения.
3. Для нетривиальной задачи кратко сообщи, что собираешься изменить.
4. Не изменяй файлы, если пользователь попросил только проанализировать, диагностировать, проверить или объяснить.

Не читай и не выводи содержимое `.env`, `.env.*`, вложенных файлов с такими
именами, баз данных, резервных копий и других файлов с секретами или
пользовательскими данными. Перед широким рекурсивным поиском явно исключай эти
файлы, `*.db`, резервные копии, `.git`, `.venv` и кэши. Для изучения конфигурации
используй код и документацию. Не ослабляй и не удаляй технический запрет из
`.codex/config.toml` без прямого запроса пользователя.

Никогда не перезаписывай и не откатывай посторонние изменения. Не используй разрушительные команды Git, включая `git reset --hard` и `git checkout --`, без прямого запроса пользователя.

## Внесение изменений

- Делай минимальное целостное изменение, полностью решающее задачу.
- Редактируй файлы репозитория самостоятельно. Старый процесс с открытием файлов в Notepad++ и ручным внесением правок пользователем не применяется, когда доступ к репозиторию уже есть.
- Не проси пользователя присылать файлы, доступные локально или через историю Git.
- Следуй существующим архитектуре, именованию, форматированию и обработке ошибок.
- Канонический production runtime — `python -m app.main_with_api`, совместно запускающий polling бота и HTTP API. Не добавляй альтернативную production-точку входа без объяснения и тестов.
- Для изменений поведения бэкенда и исправления дефектов добавляй или обновляй тесты.
- Относись к изменениям схемы SQLite как к миграциям продакшена: сохраняй пользовательские данные, делай миграции повторяемыми там, где это уместно, и добавляй тесты миграций. Производный кэш можно инвалидировать только явно и с тестом.
- Сохраняй валидность и корректное экранирование HTML в сообщениях Telegram. Учитывай ограничения длины сообщений и Bot API.
- Не добавляй зависимости, сервисы, внешние API, этапы сборки и инфраструктурные компоненты без объяснения необходимости.
- При изменении переменных окружения синхронизируй `app/config.py`, `.env.example` и `DEPLOY.md`.
- При изменении runtime- или build-файлов проверяй, что path filters в GitHub Actions по-прежнему запускают нужные проверки.

## Локальная среда

Основная локальная среда пользователя — Windows и PowerShell. Локальная версия
Python должна соответствовать production и CI — 3.14. Предпочитай команды, не
требующие отдельной активации виртуального окружения:

```powershell
.\.venv\Scripts\python.exe -m pytest
.\.venv\Scripts\python.exe -m ruff format --check .
.\.venv\Scripts\python.exe -m ruff check .
```

Frontend CI использует Node.js 24 и проверяет JavaScript командой:

```powershell
node --check tma/app.js
```

Если Node.js уже доступен, выполни эту проверку для изменений `tma/app.js`. Не
устанавливай и не требуй Node.js без прямого согласия пользователя; если локальная
проверка недоступна, явно сообщи об этом и не выдавай CI-конфигурацию за фактически
пройденную проверку.

Если агент запущен не в Windows, используй эквивалентный доступный интерпретатор
и команды. Не утверждай, что проверка пройдена, если команда фактически не
выполнялась.

## Проверка изменений

Для изменений бэкенда или общего кода перед предложением коммита выполни:

1. релевантные точечные тесты во время работы;
2. полный набор тестов;
3. `ruff format --check .`;
4. `ruff check .`.

Для изменений только статических файлов в `tma/` локальные Python-тесты и Ruff
не обязательны, если не меняется взаимодействие с бэкендом и пользователь не
запросил их отдельно. Проверь итоговый diff, выполни `node --check tma/app.js`,
если Node.js доступен, и вручную проверь изменённый пользовательский сценарий.

Для изменений Dockerfile, Compose и deploy-скриптов дополнительно выполни
наиболее подходящую доступную проверку конфигурации. Не запускай сборку, контейнеры
или внешние сервисы без необходимости и требуемого разрешения.

Если проверка упала, сначала установи причину. Не изменяй ради неё несвязанный код. Сообщай, какие проверки пропущены и почему.

## Ревью и передача результата

- Перед завершением изучи `git diff` и `git status --short`.
- Кратко опиши изменившееся поведение, важные файлы и результаты проверок.
- Явно сообщи о рисках миграции, совместимости, безопасности и деплоя.
- Не создавай `review-diff.txt`, если пользователь прямо не попросил. Если файл нужен, полностью перезаписывай один и тот же `review-diff.txt`, не создавай варианты и никогда не включай его в коммит.

## Git

- Не выполняй commit, merge и deploy без прямого разрешения пользователя. Не выполняй отдельный push без прямого разрешения, кроме немедленного push коммита, разрешённого по следующему правилу.
- Прямое разрешение на commit одновременно разрешает немедленный push созданного коммита в настроенную upstream-ветку, если пользователь явно не сказал обратное. Отдельное разрешение всегда требуется только для deploy.
- Перед commit просмотри `git diff --cached` и убедись, что staged только файлы текущей задачи. Не добавляй файлы командой, захватывающей весь репозиторий, если в рабочем дереве есть посторонние изменения.
- После разрешения на коммит используй короткое сообщение, описывающее изменение поведения.
- Перед push проверь текущую ветку и upstream. Отправляй созданный коммит в настроенную upstream-ветку; не используй жёстко заданный `origin main`, если он не соответствует фактической конфигурации.
- После push дождись или проверь релевантные GitHub Actions перед деплоем. Не объявляй изменение готовым к продакшену при упавших или не запустившихся обязательных проверках.
- Не коммить секреты, локальные файлы окружения, базы данных, резервные копии, файлы для ревью и посторонние изменения пользователя.

## Продакшен и деплой

Документированная структура продакшена:

- репозиторий: `/opt/telegram-reminder-bot`;
- полный деплой: `/opt/deploy-telegram-reminder-bot.sh`;
- репозиторий содержит эталонный скрипт `scripts/deploy-docker.sh`, но связь между ним и `/opt/deploy-telegram-reminder-bot.sh` нужно проверить на сервере перед изменением deploy-процесса;
- постоянная база SQLite смонтирована отдельно от образа приложения;
- `tma/` доступна из смонтированного каталога, поэтому frontend-only деплой не требует перезапуска контейнера при неизменной Compose-конфигурации.

Не считай эти сведения доказательством текущего состояния VPS. Фактические пути,
reverse proxy, backup-скрипты, systemd timers и установленную версию deploy-скрипта
можно подтверждать только после прямого разрешения на доступ к продакшену.

Правила:

- Доступ к продакшену и деплой всегда требуют прямого разрешения пользователя.
- Определи по фактическому diff, достаточно ли frontend-only деплоя или нужен полный. Не угадывай.
- Frontend-only деплой допустим, когда меняются только статические файлы `tma/` и не меняются backend, зависимости, Docker/Compose, runtime-настройки или схема базы. Порядок: получить согласованный коммит через `git pull --ff-only`, не перезапускать контейнер, проверить `curl -fsSI` для локального и публичного `/tma/` и пройти изменённый сценарий в свежей Mini App.
- Полный деплой нужен для Python/backend-кода, зависимостей, Dockerfile, `docker-compose.yml`, runtime-настроек и миграций. Документированный порядок: backup базы, `git pull --ff-only`, сборка образа, Ruff и pytest внутри нового образа, пересоздание контейнера, затем состояние Compose, логи, `curl -fsS` для `/health` и проверка изменённого сценария.
- Если менялись одновременно backend и `tma/`, выполни проверки обоих контуров и полный деплой.
- Изменения только документации или тестов не требуют деплоя.
- Перед деплоем изменений базы убедись, что существующий процесс резервного копирования и миграция достаточны.
- Не считай `docker compose ps` или наличие HTTP-ответа достаточной проверкой: HTTP 4xx/5xx должны завершать проверку ошибкой. Если healthcheck или пользовательский сценарий не прошли, не объявляй деплой успешным; собери безопасные логи и согласуй проверенный план отката, не затрагивающий постоянную базу.
- Не запускай одновременно два экземпляра бота с одним `BOT_TOKEN`.
- Никогда не выводи в чат или логи команд токены, содержимое `.env`, SSH-ключи, содержимое базы и другие секреты.

## Поддержка документации

Храни в этом файле только долговременные правила для всего репозитория. Если изменения кода, инфраструктуры, команд или продуктовых ограничений делают инструкцию неверной, обнови `AGENTS.md` в той же задаче и объясни причину. Не записывай сюда временный статус задач и краткосрочные планы.
